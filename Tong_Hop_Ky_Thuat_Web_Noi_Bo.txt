## II. DATABASE (THÊM BẢNG MỚI TRÊN SUPABASE)
*Không tính các bảng cũ (Categories, Services, Bookings, Booking Items). Thiết kế tối ưu cho PostgreSQL (Supabase)*

2. Bảng Khách Hàng: `customers` (Nâng cấp luồng Đăng nhập)
   - Mục đích: Lưu trữ độc lập thông tin khách vãng lai (Guest_xxx) và khách Login Google để định danh lúc checkout, theo dõi số khung giờ quay lại (History).
   - `id` (UUID/String, Primary Key)
   - `full_name` (String)
   - `email` (String, Unique)
   - `phone` (String, Nullable)
   - `is_google_auth` (Boolean): True nếu khách đăng nhập bằng Google.
   - `avatar_url` (String)
   - `created_at`, `last_visited_at` (Timestamptz)

3. Bảng Cấu hình Đánh Giá Động: `admin_question_configs`
   - Mục đích: Lễ tân/Quản lý có thể thay đổi câu hỏi Live Checklist dễ dàng trên DB mà Frontend không phải hard code.
   - `id` (UUID, Primary Key)
   - `question_text` (String): ND câu hỏi (VD: "Nhân viên bấm điện thoại?")
   - `question_type` (Enum): Loai "Live" hoặc "Post"
   - `is_active` (Boolean): Cho phép bật/tắt câu hỏi hiển thị cho khách.
   - `created_at`, `updated_at` (Timestamptz)

2. Bảng Đánh Giá Tức Thời: `live_checkpoints`
   - Mục đích: Nơi tiếp nhận Feedback ngay khi làm dịch vụ (Phát tín hiệu Real-time báo Admin).
   - `id` (UUID, Primary Key)
   - `booking_id` (UUID, Foreign Key -> `bookings(id)`): Map với phòng/khách hàng nào đang complaint.
   - `question_id` (UUID, Foreign Key -> `admin_question_configs(id)`)
   - `is_issue` (Boolean): Đánh dấu có bấm vô lỗi (True/False).
   - `created_at` (Timestamptz)

3. Bảng Feedback Tổng Hợp & Tip: `customer_feedbacks`
   - Mục đích: Thu thập Data chốt ngày (Rating + Tiền boa).
   - `id` (UUID, Primary Key)
   - `booking_id` (UUID, Foreign Key -> `bookings(id)` - Unique: Mỗi bill đánh 1 lần)
   - `customer_email` (String)
   - `technician_code` (String): ID nhân viên đấm bóp (Lấy từ booking đổ qua).
   - `rating_score` (Int2): Điểm số (1-5 sao/icon).
   - `review_comment` (Text): Góp ý chữ.
   - `tip_amount` (Numeric): Tiền Típ (Mặc định 0).
   - `created_at` (Timestamptz)

## III. BACKEND (THIẾT KẾ & TÍCH HỢP NEXT.JS API / SUPABASE SDK)
*Các Endpoint và Realtime Channel cần dựng*

1. API Xác thực Khách & Sync Data - `/api/auth/customer-sync` (Post-Login Hook)
   - Nhiệm vụ: Xử lý ngay sau khi Supabase Trả về Token Google thành công.
   - Payload: Google Profile Data (`email`, `name`, `avatar`).
   - Xử lý Upsert: Bắn vào bảng `customers`. Nếu email đã tồn tại -> Update `last_visited_at`. Nếu chưa -> Tạo mới khách hàng (`is_google_auth = true`).

2. API Filter Dịch Vụ - `/api/services/filter` (Hoặc gọi qua Supabase JS Client)
   - Nhiệm vụ: Xử lý tìm kiếm dựa trên bảng JSON `comboTags`.
   - Payload đầu vào: `["body", "wash"]`.
   - Xử lý: Select những `services` có `comboTags` chứa điều kiện mảng truyền vào (Dùng PostgreSQL JSON Operator `?&` hoặc `@>`).

2. API Push Đánh Giá Tức Thời - `/api/feedbacks/live`
   - Nhiệm vụ: Lưu dòng click checklist vào `live_checkpoints`.
   - Không cần trả Data lớn, Code chuẩn REST: Trả Status 2xx để FE đóng popup thông báo là đủ.
   - Cấu hình Broadcast/Channel Supabase: Kích hoạt Insert Record Trigger để kênh `postgres_changes` bắn notifi hỏa tốc cho Giao diện Lễ Tân nằm vùng.

3. API Yêu cầu Cứu Gọi (Đổi Nhân Viên) - `/api/bookings/switch-staff`
   - Nhiệm vụ: Xử lý khi khách bấm nút "Đổi KTV".
   - Payload: `booking_id`, `reason`.
   - Update bảng `bookings`: `status` = 'switch_requested' và `is_switched` = true (Tránh spam click).
   - Bắn Pusher / Supabase Realtime cho Quản Lý.

4. API Chốt Feedback Cuối Đơn - `/api/feedbacks/post`
   - Nhiệm vụ: Ghi số sao vả Tiền Tip. Cập nhật Status hoàn chỉnh hóa đơn.
   - Gom 2 lệnh (Transaction):
     + Lệnh 1: Insert sang `customer_feedbacks` (Rating + Tip).
     + Lệnh 2: Update `bookings`.`status` = 'completed'. (Việc gán Status completed rất quan trọng vì nhờ nó Màn hình KTV mới tải được Feedback của chính họ về để xem).

5. Setup Real-time & Rules (Database Triggers & Policy)
   - Cảnh báo RLS (Row Level Security): Tạo 1 policy khóa Read đối với KTV trên cụm bảng Feedback. Chỉ cấp quyền SELECT đối với record nào có `status` của Booking join theo bằng 'completed'. Mức Policy này cực quan trọng để bảo vệ KTV và Trải nghiệm khách hàng nội bộ.
